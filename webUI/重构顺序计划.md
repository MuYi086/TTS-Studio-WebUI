# WebUI 重构顺序计划

## 目标

在 **不影响现有 `index.html` 执行逻辑** 的前提下，新建 `webUI/` 作为新的 Vue 工程目录，按渐进增强方式重构当前单文件应用。整个过程以旧 `index.html` 为行为对照物，以 `P1-最小回归验收.md` 为最低验收门槛。

## 总原则

1. 旧 `index.html` 在切换主入口前持续保留，不直接改成半新半旧状态。
2. 先迁移数据结构、纯逻辑和副作用边界，再迁移界面。
3. 每一阶段都必须可运行、可回归、可回退。
4. 本轮只做 `P2` 范围内的 Vue 工程化重构，不顺手改协议、改 schema（工程结构版本）、改 UI 视觉。

## P0：冻结基线与搭建迁移护栏

### 目标

先把“现在到底什么算正确”固定下来，避免后续迁移时凭印象重写。

### 范围

- 基线固化
- 回归入口固化
- `webUI/` 目录初始化约束

### 任务

1. 建立 `webUI/` 目录，明确它是新工程根目录，旧实现继续留在仓库根目录。
2. 把以下文件定义为当前行为对照物：
   - `index.html`
   - `project-storage.js`
   - `voice-design.js`
3. 以 `P1-最小回归验收.md` 和 `fixtures/p1-minimal-regression-project-v3.json` 固化最小回归场景。
4. 补一份迁移清单，明确 5 个主域：
   - 模型配置
   - 音色资源库
   - 音效 / `BGM` / 滤波器资源库
   - 脚本工作台
   - Prompt 管理
5. 明确兼容性红线：
   - 本地键名 `UnitaleDB`、`project`、`assets`
   - 工程文件 `kind/schemaVersion/version`
   - `audioAssetKey`、`bgImageAssetKey`、`assetKey`
   - `WAV/SRT/MP4` 时间轴与播放语义
6. 规定迁移期间禁止项：
   - 不直接在新组件里复制旧的超大函数
   - 不让组件直接访问 `fetch`、`IndexedDB`、`localStorage`
   - 不删除旧页面

### 输出

- `webUI/重构顺序计划.md`
- 可重复执行的 P1 手动回归入口
- 兼容性红线清单

### 验收

1. 新成员只看仓库文档，就能知道旧页面哪些行为必须保持不变。
2. `node scripts/validate-p1-regression-fixture.mjs` 可作为固定入口。
3. 后续任何迁移任务都能明确归到某个域和某个阶段。

## P1：先搭新工程骨架，再迁移稳定边界

### 目标

在 `webUI/` 内先建立可运行的 Vue/Vite 工程，把最稳定、最容易验证的部分先拆出来，但暂不迁移高风险音频编排和导出链路。

### 范围

- 工程骨架
- 领域模型与纯函数
- 存储 / 协议服务层
- Store（状态仓库）骨架
- 低风险页面外壳

### 执行顺序

1. 初始化 `Vite + Vue 3 + TypeScript`。
2. 接入 `Pinia`，补基础命令。
3. 建立基础目录：
   - `src/app`
   - `src/domain`
   - `src/services`
   - `src/stores`
   - `src/components`
4. 先迁移 `project-storage.js` 里的纯逻辑到 `TypeScript`：
   - `normalizeProjectEnvelope`
   - `stripRuntimeProjectEnvelope`
   - `createAssetKey`
   - 各类脚本块工厂函数
   - 角色 / 音色匹配辅助函数
5. 用固定 `fixture` 和手动对照方式验证上述纯逻辑迁移结果。
6. 再封装副作用层：
   - `projectRepository`
   - `assetRepository`
   - `preferenceRepository`
   - `llmClient`
   - `ttsClient`
   - `voiceDesignClient`
7. 建立最小 Store 切分：
   - `settingsStore`
   - `libraryStore`
   - `projectStore`
   - `playbackStore`
   - `jobsStore`
8. 只迁移低耦合界面外壳和基础编辑能力：
   - 顶部页签
   - 模型配置页
   - Prompt 管理页
   - 不依赖音频引擎的基础表单与列表

### 输出

- `webUI/` 下可启动的新工程
- 纯逻辑迁移结果与手动校验记录
- 存储与协议服务层
- Store 边界骨架
- 能展示基础页签和部分低风险页面的新 UI

### 验收

1. 新工程可以独立 `dev/build/typecheck`。
2. `fixture` 能被新领域模块正确解析，结果与旧实现一致。
3. 组件中不再直接出现 `fetch`、`indexedDB`、`localStorage`。
4. 低风险页面在新工程中可用，但旧 `index.html` 仍是默认可用实现。

## P2：迁移核心工作流并完成可切换版本

### 目标

迁移脚本工作台、音频预览、批量生成、顺序播放、导入导出与视频导出，形成一个可通过 P1 回归的完整新工程。

### 范围

- 资源库复杂交互
- 脚本编辑核心流程
- AI 分析 / 音色设计 / TTS 生成
- 预览播放与批量任务
- 完整导入导出链路
- 新旧入口切换准备

### 执行顺序

1. 迁移资源库复杂能力：
   - 音色上传与本地恢复
   - `SFX/BGM` 上传、试听、裁剪
   - 滤波器编辑
   - 情绪预设管理
2. 迁移脚本工作台基础编辑：
   - 多脚本标签
   - 角色绑定
   - `dialogue` / `bgm` / `bgImage` 三类块编辑
   - 选中行、插入块、排序、删除
3. 迁移 AI 工作流：
   - `LLM` 深度分析
   - 角色音色分析
   - 音色生成
   - 导入后角色与音色同步
4. 迁移音频运行时：
   - 单行台词生成 / 清空
   - 波形绘制
   - 裁剪换算
   - 试听与停止
5. 迁移流程型能力：
   - 顺序播放
   - 批量生成
   - 舞台背景预扫描
   - `BGM` 淡入淡出
6. 迁移完整导入导出：
   - 导入完整工程
   - 导出完整工程
   - 导出 `SRT`
   - 导出 `WAV`
   - 生成 `MP4`
7. 补手动回归清单：
   - 导入样例工程 -> 刷新 -> 断言恢复
   - 清除首条台词 -> 重新生成 -> 刷新 -> 断言音频恢复
   - 导出 `SRT/WAV/MP4`
8. 当新工程完整跑通后，再切换默认入口；旧 `index.html` 转为归档参考。

### 输出

- 可完整使用的新 `webUI` 工程
- 覆盖核心链路的手动回归清单
- 与旧实现兼容的新导入导出与本地恢复链路

### 验收

1. `P1` 固定样例工程可在新工程中完成导入、刷新恢复、导出 `SRT`、导出音频、生成视频。
2. 模型配置、资源库、脚本编辑、AI 分析、音色生成、台词生成、顺序播放全部可用。
3. 新工程成为默认入口前，能与旧页面逐项对照验证。
4. 旧实现仍保留为行为归档，直到切换完成。

## 本轮建议的实际开工顺序

1. 先在 `webUI/` 初始化工程骨架。
2. 第一批代码优先做 `domain/project` 和 `domain/script`。
3. 第二批再做 `services/storage` 与 `services/providers`。
4. 第三批才开始接 `Pinia` 和低风险页面。
5. 脚本工作台、音频引擎、导出链路统一放到 `P2` 后半段。

## 暂不做

- Provider（模型提供商）统一抽象升级到 `P3`
- `Vitest` / `Playwright` 自动化测试
- UI 改版
- 新功能扩展
- 路由化改造
- 服务端中转

## 一句话结论

先把 `webUI/` 变成一个“有骨架、有类型、有回归”的新工程，再把 `index.html` 里最重的脚本编辑、音频编排和导出链路按顺序迁过去，最后才切主入口。
