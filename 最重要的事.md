# 后续最重要的事

这个项目下一阶段最重要的事，是把当前单文件可用原型，升级成“不会丢工程、能稳定发布、能继续加模型”的工程化版本。

目前核心功能已经比较完整：LLM 分析、角色音色、TTS、音效、BGM、滤波器、IndexedDB 存档、WAV/SRT/MP4 导出都已经集中在 `index.html` 中。但 `index.html` 已经超过 5700 行，UI、状态、存储、音频引擎、视频导出和模型请求都混在一起。继续直接堆功能，会让存档、导入导出、批量生成和视频导出这些关键链路越来越容易出问题。

## 建议优先级

1. **P0：存档与资产可靠性**
   先明确工程 JSON、IndexedDB 素材、音频 URL、背景图片、脚本多标签的数据结构，建立版本号和迁移逻辑。这个工具最怕用户制作数小时后工程打不开或素材丢失。

2. **P1：建立最小回归验收**
   准备固定样例工程，覆盖导入工程、恢复素材、生成/清空台词音频、导出 WAV、导出 SRT、生成 MP4。先做手动检查清单也可以，后续再升级到 Playwright（浏览器端自动化测试）或 Vitest（单元测试框架）。

3. **P2：重构为 Vue/Vite 工程**
   不建议直接重写。应该在验收链路保护下逐步拆模块：`storage`（存档和 IndexedDB）、`llm`（Prompt 和 JSON 解析）、`tts`（TTS 和音色设计接口）、`audio`（播放、混音和导出）、`video`（MP4 导出）、`components`（界面组件）。

4. **P3：模型提供商抽象**
   现在 `voice-design.js` 已经把 Qwen 和 MIMO 做成配置，这是正确方向。下一步应该把 LLM、TTS、voice design（音色设计）都统一成 provider（模型提供商）配置，避免每加一个模型都修改主文件逻辑。

5. **P4：再做 UI 和新功能**
   等可靠性和工程结构稳定后，再考虑批量任务队列、失败重试、后台合成状态、云端中转、模板市场、素材库搜索等功能。

## 一句话结论

先保工程不丢、结果可复现、代码可拆，再继续扩功能。
