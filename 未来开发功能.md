# 未来开发功能

本文件用于整理 `P2` 之后的后续开发方向，避免口头讨论和正式规划混在一起。

## 规划边界

- 当前仓库里，正式写入文档的后续阶段只有 `P3` 和 `P4`。
- `P3` 来源于 [webUI/重构顺序计划.md](webUI/重构顺序计划.md) 的“暂不做”条目，以及 [最重要的事.md](最重要的事.md) 的后续优先级说明。
- `P4` 来源于 [最重要的事.md](最重要的事.md)。
- 当前仓库没有正式定义 `P5`。
- 因此，下面的 `P5` 属于“建议阶段”，不是当前已经承诺的正式里程碑。

## P3：模型提供商抽象

### 目标

把 `LLM`、`TTS`、`voice design`（音色设计）统一抽象为 provider（模型提供商）体系，降低新增模型和切换模型的改动成本。

### 要解决的问题

- 现在不同能力的接入方式还不够统一。
- 新增一个模型时，容易改到主流程、主页面或多个调用点。
- 模型配置、鉴权、能力声明和请求参数缺少统一约束。

### 预期能力

- 统一 provider 配置结构：
  - 基础地址
  - 模型名
  - 鉴权信息
  - 超时与重试策略
  - 支持的能力范围
- 统一 provider 注册表：
  - `LLM` provider
  - `TTS` provider
  - `voice design` provider
- 统一适配器接口：
  - 文本分析调用
  - 角色音色分析调用
  - 音色生成调用
  - 台词合成调用
- `UI` 改为面向 provider 配置，而不是面向具体实现写死逻辑。
- 新增模型时，优先通过“新增配置 + 新增适配器”完成，而不是修改主工作流。

### 验收口径

- 新增一个模型提供商时，不需要改动脚本工作台主流程。
- 同一类能力可以在多个 provider 之间切换。
- 模型配置页能够清晰区分不同 provider 的可用能力。

## P4：UI 优化与新功能扩展

### 目标

在 `P2` 可验收、`P3` 架构稳定后，再进入用户体验和功能扩展阶段。

### 适合放在 P4 的方向

- `UI` 改版与交互优化
- 批量任务队列
- 失败重试
- 后台合成状态展示
- 云端中转
- 模板市场
- 素材库搜索

### 设计原则

- 不在 `P2` 验收前提前做大规模视觉改版。
- 不为了新功能破坏现有工程兼容性、导入导出链路和本地恢复能力。
- 新功能优先建立在已完成的 provider、存储和音频运行时边界上。

### 适合拆分的子主题

- 工作流效率：
  - 批量任务队列
  - 失败重试
  - 后台状态可视化
- 创作体验：
  - 更好的脚本编辑体验
  - 更清晰的资源库管理
  - 更强的模板复用能力
- 交付能力：
  - 云端中转
  - 更稳定的长任务处理

## P5：建议阶段，当前未正式立项

### 定位说明

当前仓库没有正式定义 `P5`。如果后面继续排期，建议把“自动化质量保障与服务化能力”作为 `P5`。

### 建议目标

让项目从“可用的本地工具”进一步升级为“可持续迭代、可稳定发布、可扩展部署”的产品级工程。

### 建议范围

- `Vitest` 单元测试
- `Playwright` 浏览器端回归测试
- 核心链路验收脚本自动化
- 路由化改造
- 服务端中转与任务编排
- 长任务状态持久化
- 错误观测与日志诊断

### 建议价值

- 降低后续改 UI、换模型、扩功能时的回归风险。
- 让批量生成、导出视频、恢复工程这些高风险链路可以重复验证。
- 为未来云端任务、多人协作或服务化部署预留基础能力。

## 一句话结论

当前正式后续规划可以理解为：

- `P3`：先把模型接入方式抽象干净。
- `P4`：再做 UI 优化和新功能扩展。
- `P5`：如果继续往前推进，建议补自动化质量保障和服务化能力，但这一步目前还不是正式承诺。
